`pattern` is an array of **positions**. Each position is a JSON object that must
be satisfied by one onset of the [driving
feature](/docs/query-schema#the-driving-feature) before the search moves on
to the next position and the next onset. A 3-position pattern only matches where
3 *consecutive* onsets each satisfy their own position, in order.

```json
"pattern": [
    { "deg": "3" },
    { "deg": "2" },
    { "deg": ["1", "3"] }
]
```



## Values

A key's value can be a single string, or an array of strings. An array means
"any of these". The position is satisfied if the actual value equals **any**
entry in the list:

```json
{ "deg": ["1", "3"] }
```

matches wherever the scale degree is `1` **or** `3`.

A plain JSON boolean is accepted too and coerced to the strings
`"true"`/`"false"`, so `{ "fermata": true }` and `{ "fermata": "true" }` are
equivalent.



## Wildcards

The literal value `"*"`, anywhere in a key's value/list, means "match anything".
The check for that key is skipped entirely. It's how you require an onset to
*exist* without constraining what it is:

```json
"pattern": [
    { "fermata": "true" },
    { "kern": "*" }
]
```

finds every fermata-marked note followed immediately by *any* next note, i.e.
every place a fermata isn't the very last note of a voice.



## Checking several features at once

A position isn't limited to one key. Every key present must match (an implicit
AND across keys), and a key doesn't have to be the driving feature. It can
reference *any* feature, looked up on the same line the driving feature's onset
landed on:

```json
{
    "feature": "deg",
    "pattern": [
        { "deg": "3", "duration": "4" },
        { "deg": "2", "duration": "4" },
        { "deg": ["1", "3"], "duration": "*", "fermata": "true" }
    ]
}
```

This is driven by `deg` (so it walks scale-degree onsets), but each position
also constrains `duration` (Humdrum recip notation: `"4"` = quarter note, `"8"`
= eighth, etc.) and, on the last position, `fermata`. Mixing features like this
is how you combine a melodic/harmonic shape with rhythmic or expressive
constraints in a single query.



## Negating a feature

Prefix any key with `!` to require that it does **not** match. The negation
applies to the whole value list at once, not to each entry on its own. An
OR-list under `!` becomes "none of these", not "not all of these":

```json
{ "!deg": ["3", "5"] }
```

matches whenever the scale degree is **neither** 3 **nor** 5. Negation works
uniformly on any key in a pattern: the driving feature, a cross-referenced
feature, `duration`, `fermata`, or the [quality-aware `mint`/`fb`
matching](/docs/features) described on the feature pages.

A practical use, catching deceptive cadences: scale degree 5 followed by a
fermata-marked note that is not the first scale degree:

```json
{
    "feature": "deg",
    "voice": "bass",
    "pattern": [
        { "deg": "5" },
        { "!deg": "1", "fermata": "true" }
    ]
}
```



## Putting it together

A position, in full generality, is:

```json
{
    "<feature or !feature>": "<value>" | ["<value>", "<value>", ...] | "*",
    "<feature or !feature>": "..."
}
```

repeated once per position in the `pattern` array. See
[Features](/docs/features) for what a "value" looks like for each specific
feature, and [Example Queries](/docs/examples) for complete, runnable queries
built this way.
